How to publish your benchmark

The disk-duel script runs a 19-test fio-based suite on one or two drives, generates a local HTML report, and (optionally) uploads the results here.

{% set run_only = 'curl -fsSL https://raw.githubusercontent.com/digitalhen/disk-duel/main/disk_duel.py -o /tmp/disk_duel.py && python3 /tmp/disk_duel.py' %} {% set run_upload = run_only + ' --upload' %} {% set fio_brew = 'brew install fio' %}

1. Install fio

The script needs fio for the actual benchmark.

macOS 
${{ fio_brew }}

Linux: sudo apt-get install fio (Debian/Ubuntu) or your distro's equivalent.

2. Run & publish

One line. Downloads the script, launches the interactive drive menu, and uploads at the end.

run & publish 
${{ run_upload }}

Don't want to upload? Drop --upload. The HTML/JSON report still lands in your current directory.

How submissions are protected

The submit endpoint is unauthenticated — the script is open source so an embedded API key would be theater. Instead the script does a small proof-of-work (~1–2 s on a recent laptop) before posting; the server also rate-limits per machine (one upload / 60 s) and per IP. Replays of an old payload are rejected because timestamps must be fresh within ±5 min.

Privacy

URLs use opaque hashids slugs. Your hardware serial is hashed (sha256) on the server before storage and is never returned in any API response or rendered in any page.

What's stored

  • Hardware: machine model, chip, RAM (no serial number).
  • Drive: media name (e.g. WD_BLACK SN850X 4000GB), bus protocol, internal/external flag, size, enclosure if external.
  • Per test: the primary metric (MB/s, IOPS, or p99 latency µs) plus all the bandwidth/IOPS/latency breakdowns fio emits.
  • Raw payload: the full JSON that the script wrote alongside the local HTML report, kept for future re-rendering.
